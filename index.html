Distributed trust for AI workflows

Split AI context across providers so no single model sees the whole.

Prompt Sharding is a phone-first security protocol for multi-model workflows. It uses a GitHub ledger, signed JSON envelopes, and agent inboxes so Perplexity, Claude, and ChatGPT can collaborate without any one provider receiving the complete picture.

How it works

The ledger is the trust boundary.

Instead of giving one AI provider the entire conversation, Prompt Sharding assigns each provider a role-specific slice. The full result is assembled only in the ledger, where every move is auditable.

1

Define the task

A contract review, code review, research run, or game turn enters the ledger as a signed envelope.

2

Split into shards

The router uses shards.json to assign each model the minimum necessary context for its role.

3

Assemble receipts

Each agent signs its output. The ledger combines the results without ever sending the full prompt back to any one provider.

Immediate uses

Useful on day one.

This is not a hypothetical research toy. The first product is already clear: personal and team workflows where provider segmentation matters right now.

Multi-model code review

Route architecture to one model, security to another, and tests to a third. No single provider sees the whole review context.

Personal research pipelines

One model gathers sources, another checks contradictions, another writes the final brief. The protocol coordinates the handoffs.

Async game engines

Turn-based games can live as git history. Each move is a signed envelope, each new state is regenerated and logged without streaming.

Security posture

Strong today, honest about the edges.

Prompt Sharding already improves privacy against single-provider exposure. v1 does not claim perfect security; it clearly defines what it protects and what still depends on the GitHub ledger.

Protects nowWhy it mattersStill on you in v1
Provider breach containmentA compromised model vendor sees only one shard, not the total task.Your GitHub token and repo access must be protected.
Training-data minimizationProviders only receive a role-specific fragment, which lowers exposure.The ledger is still plaintext until encrypted storage lands in v2.
Prompt isolationBad instructions or injected context stay contained to one role instead of poisoning the whole workflow.The router and envelope policies still need stronger enforcement.

v1 security note

Your GitHub Personal Access Token is the key to the ledger. Treat it like a password. Store it in a password manager, keep repo permissions tight, and never paste it into prompts or markdown files. Prompt Sharding protects against single-provider overexposure, but the ledger is only as secure as your GitHub account.
